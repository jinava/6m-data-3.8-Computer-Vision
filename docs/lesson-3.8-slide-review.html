① High priority — conceptual error

SLIDE 5 · “Today's Three Questions” · Q3

HighTransfer learning is gated on “domain close to ImageNet”

The slide states: “When should we reach for transfer learning? When data is small AND the domain is close to ImageNet.”

This is misleading and internally inconsistent with the rest of the deck. Transfer learning is the right default for small data regardless of domain closeness — domain closeness decides how much you fine-tune (head-only feature extraction vs. unfreezing layer4), not whether to use transfer learning at all.

Evidence the deck contradicts itself

  • Slide 20 (“The Honest Result”) shows transfer learning winning on Fashion-MNIST (0.832 fine-tuned vs 0.787 from scratch) — and that is explicitly the far domain (“big domain gap”).
  • Slide 23 (“Method Cheat Sheet”) says “< 1K images per class → Transfer learning” with no domain caveat.
  • lesson.md states transfer is “almost always the right starting point” under ~1K/class.

A sharp student will hit slide 20 and notice it directly refutes Q3.

Fix: reword to decouple the two decisions, e.g. “Reach for transfer learning whenever data is small; the domain gap then decides head-only feature-extraction vs. fine-tuning the last conv block.”

② Medium — mislabel that conflicts with the notebook

SLIDE 13 · “Classical Edge Detection Kernels”

MediumSobel/Prewitt Gx/Gy orientation labels are swapped

The kernel Gx = [[-1,0,1],[-2,0,2],[-1,0,1]] is labelled “Horizontal Edge Detection.” But this kernel computes the horizontal gradient and therefore responds to vertical edges — and the slide's own thumbnail under it shows a vertical white line. A student reading “Horizontal Edge Detection” concludes the opposite of what the picture shows. The same swap applies to Gy (“Vertical Edge Detection”, but it detects horizontal edges).

Conflicts with the course notebook

In 02_convolutions_intuition.ipynb (Extension E2), this exact kernel is named sobel_v and correctly described as a vertical edge detector (“reacts to left-to-right brightness changes, so it lights up VERTICAL edges”). Slide and notebook therefore disagree on the same matrix.

Fix: label by what the kernel detects, e.g. “Gx (horizontal gradient → detects vertical edges)”, so the slide, its own thumbnail, and the notebook all agree.

SLIDES 4, 11, 12 · diagrams

LowChannels-last notation vs. PyTorch channels-first

Slides use channels-last ordering (28×28×1, 224×224×3 on slides 4 & 12; 24x24x32 on the CNN-block diagram). The entire course code is PyTorch = channels-first: (1,28,28), (B, C, H, W), as stated in the notebooks and lesson.md.

Fix: add a one-line note that slide diagrams use the H×W×C convention while PyTorch tensors are C×H×W — otherwise students trip over the mismatch when they reach the code.
SLIDE 11 · “Question 2” pipeline

Low“FC Layer → Linear” is redundant; Flatten missing

The text pipeline lists … → Fully Connected (FC) Layer → Linear as two sequential steps, but an FC layer is a Linear layer, and Flatten (named in the box header) is absent from the chain.

Fix: Conv2d → ReLU → MaxPool2d → Flatten → Linear (FC).
SLIDES 11 vs 12 · “The CNN Block”

LowTwo different architectures shown without comment

Slide 11 uses Conv2d(kernel=3, padding=1) (size-preserving); slide 12's diagram uses 5×5 convs with no padding (28→24→12→8→4). Both are valid, but the silent switch in kernel size/padding can confuse. A one-line note that these are two illustrative configurations would help.

SLIDE 23 · “Method Cheat Sheet”

LowPer-class vs. total counts at the same threshold

“10K+ small images, 10 classes” (≈1K/class) sits directly above “< 1K images per class”, mixing total and per-class counts near the same boundary.

Fix: express both rows per-class for a clean threshold.
